# Protocol Telemetry - Refactored Architecture

## Overview

This document outlines the modular, production-grade refactoring of the Protocol Telemetry Streamlit application. The monolithic `app.py` has been split into a multipage application with clear separation of concerns.

## Directory Structure

```
fitness_tracker/
├── app.py                      # Home page / Entry point
├── database.py                 # Supabase connection (cached)
├── visualizations.py           # Pure Python plotting functions
├── requirements.txt            # Dependencies
├── pages/
│   ├── 1_Gym.py               # Mechanical Tension logging
│   ├── 2_Swim.py              # Cardiovascular logging
│   ├── 3_Nutrition.py         # Biochemical inputs logging
│   ├── 4_Body.py              # Anthropometric data logging
│   ├── 5_Recovery.py          # Systemic fatigue & recovery logging
│   └── 6_Dashboard.py         # Analytics & forecasting (controller page)
└── .streamlit/                # (Auto-generated by Streamlit)
```

## Architecture & Design Principles

### Rule 1: Database Isolation ✓

**File:** [database.py](database.py)

- Centralized Supabase client initialization
- Uses `@st.cache_resource` decorator to prevent re-initialization on page reloads
- Single import point across all pages: `from database import supabase`
- Maintains connection pooling efficiency

```python
# All pages use:
from database import supabase
```

### Rule 2: Native Multipage Routing ✓

**Removed:** `st.tabs()` logic

**Implemented:**
- Streamlit's native multipage routing via `pages/` directory
- Page numbering (`1_Gym.py`, `2_Swim.py`) controls sidebar order
- No tab state management needed
- Cleaner, more maintainable structure

### Rule 3: Visualization Decoupling ✓

**File:** [visualizations.py](visualizations.py)

All visualization functions are **pure Python** with zero Streamlit dependencies:

1. **`create_consistency_heatmap(gym_df, swim_df, days=180)`**
   - GitHub-style activity heatmap (Rest/Gym/Swim/Both)
   - Returns: `go.Figure`

2. **`create_daily_tonnage_chart(merged_gym)`**
   - Daily work capacity bar chart
   - Returns: `go.Figure`

3. **`create_split_balance_radar(merged_gym)`**
   - Radar chart of split distribution
   - Returns: `go.Figure`

4. **`create_weight_forecast(body_df)`**
   - OLS regression + 30-day forecast
   - Returns: `go.Figure`

5. **`create_cardio_telemetry(swim_df, rec_df)`**
   - Dual-axis chart (swim pace vs RHR)
   - Returns: `go.Figure`

6. **`create_metabolic_intake_chart(nut_df)`**
   - Calorie intake with 7-day rolling average
   - Returns: `go.Figure`

7. **`get_max_1rm(merged_gym, exercise_name)`**
   - Extracts max 1RM for a given exercise
   - Returns: `str` (formatted)

### Rule 4: Dashboard Assembly ✓

**File:** [pages/6_Dashboard.py](pages/6_Dashboard.py)

The dashboard acts as the **controller**:

1. **Data Extraction** → Queries Supabase
2. **Transformation** → Creates pandas DataFrames
3. **Type Casting** → Converts to numeric types
4. **Complex Calculations** → Tonnage, 1RM estimates
5. **Visualization Calls** → Imports and calls functions from `visualizations.py`
6. **Rendering** → Uses `st.plotly_chart()` for display

```python
# Example workflow:
from database import supabase
import visualizations as viz

# Extract
gym_resp = supabase.table("gym_logs").select(...).execute()
gym_df = pd.DataFrame(gym_resp.data)

# Call visualization function (returns go.Figure)
fig = viz.create_consistency_heatmap(gym_df, swim_df)

# Render
st.plotly_chart(fig, use_container_width=True)
```

### Rule 5: No Code Left Behind ✓

All functional code preserved across files:

| Component | Location | Status |
|-----------|----------|--------|
| `insert_exercise_data()` helper | [pages/1_Gym.py](pages/1_Gym.py) | ✓ Preserved |
| `pd.to_numeric()` type casting | [pages/6_Dashboard.py](pages/6_Dashboard.py) | ✓ Preserved |
| Brzycki 1RM formula | [pages/6_Dashboard.py](pages/6_Dashboard.py) | ✓ Preserved |
| Rest day logic | Each logging page | ✓ Preserved |
| Toast/error messages | All pages | ✓ Preserved |

## File-by-File Breakdown

### [app.py](app.py) - Home Page

**Purpose:** Welcome screen and navigation hub

**Features:**
- Page title and description
- Navigation guide to each section
- Quick reference table of tracked metrics

### [database.py](database.py) - Connection Layer

**Key Export:**
```python
supabase  # Cached Supabase client
```

**Why cached?** Prevents token refresh on every page reload, maintaining connection pooling.

### [visualizations.py](visualizations.py) - Plotting Logic

**Key Principle:** Pure functions, no Streamlit

**All functions take pandas DataFrames and return Plotly Figures**

Example:
```python
def create_weight_forecast(body_df: pd.DataFrame) -> go.Figure:
    """Takes a DataFrame, returns a Figure"""
    # No st.subheader(), st.plotly_chart(), etc.
    # Pure math and Plotly object construction
    return fig
```

### [pages/1_Gym.py](pages/1_Gym.py) - Gym Logging

**Features:**
- Rest day toggle
- Dynamic exercise selection by split
- Multi-set logging with weight, reps, RIR
- `insert_exercise_data()` helper function
- Supabase insertion (parent + child records)

### [pages/2_Swim.py](pages/2_Swim.py) - Swim Logging

**Features:**
- Rest day toggle
- Swim type selection (Endurance/Sprints)
- Duration & distance input
- Auto lap time calculation
- Supabase insertion

### [pages/3_Nutrition.py](pages/3_Nutrition.py) - Nutrition Logging

**Features:**
- Calorie & macronutrient input
- Water intake tracking
- Simple form-based logging

### [pages/4_Body.py](pages/4_Body.py) - Body Metrics

**Features:**
- Daily weight logging
- Optional bi-weekly tape measurements
- Supabase insertion

### [pages/5_Recovery.py](pages/5_Recovery.py) - Recovery Logging

**Features:**
- Sleep duration & RHR input
- 4x pain discomfort sliders (shoulder, elbow, wrist, lower back)

- Supabase insertion

### [pages/6_Dashboard.py](pages/6_Dashboard.py) - Analytics Hub

**Workflow:**
1. Refresh button triggers full ETL pipeline
2. Extracts all tables from Supabase
3. Transforms to DataFrames
4. Performs type casting & calculations
5. Calls visualization functions
6. Renders charts and KPIs

**Displayed Sections:**
- Peak Mechanical Outputs (1RM metrics)
- Activity Consistency Heatmap
- Training Consistency (daily tonnage)
- Split Balance (radar)
- Body Recomposition Forecast
- Metabolic & Cardiovascular Telemetry
- Summary Statistics

## Running the Application

### Local Development

```bash
# Install dependencies
pip install -r requirements.txt

# Run Streamlit
streamlit run app.py
```

Streamlit will automatically serve the multipage app at `http://localhost:8501`

**Page Sidebar Order** (auto-generated):
1. Home (app.py)
2. Gym
3. Swim
4. Nutrition
5. Body
6. Recovery
7. Dashboard

### Deployment

For cloud deployment (Streamlit Cloud, AWS, etc.):
- Ensure `secrets.toml` contains Supabase credentials
- Directory structure will be preserved
- Multipage routing works identically

## Benefits of This Refactoring

| Aspect | Before | After |
|--------|--------|-------|
| **Code Organization** | 500+ lines in one file | Modular files by feature |
| **Testability** | Hard to unit test | Easy to test viz functions |
| **Reusability** | Tightly coupled | Pure functions in visualizations.py |
| **Maintenance** | Error-prone tab switching | Native Streamlit routing |
| **Scalability** | Adding features clutters file | Add new page easily |
| **Performance** | Connection re-init on page change | Cached connection pool |
| **Readability** | Dense monolith | Clear separation of concerns |

## Best Practices Applied

✅ **DRY Principle:** Visualization functions extracted and reused
✅ **Single Responsibility:** Each page handles one domain
✅ **Caching:** Connection stored with `@st.cache_resource`
✅ **Error Handling:** Try-catch on Supabase operations
✅ **Type Conversion:** Explicit `pd.to_numeric()` usage
✅ **Documentation:** Docstrings on all functions
✅ **Naming Convention:** Descriptive file/function names

## Troubleshooting

### Connection Issues
- Verify `secrets.toml` has `supabase.url` and `supabase.key`
- Check network connectivity to Supabase

### Missing Data on Dashboard
- Ensure at least one entry exists in each table
- Click "Refresh Data" button to pull latest

### Visualization Errors
- Check DataFrame columns match function signatures
- Verify numeric type casting is successful
